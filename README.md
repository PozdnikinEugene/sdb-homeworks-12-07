### Задание 2
#### Вертикальный шардинг (разделение по столбцам/сущностям)
Суть: разные таблицы (или группы таблиц) живут в разных физических базах данных/инстансах. Это снижает нагрузку на один сервер за счёт разделения разных типов нагрузки (например, «тяжёлые отчёты по книгам» не тормозят «быстрые операции с пользователями»).

##### Как разбить для трёх таблиц:

- Shard A (Users): таблица 'users' + все связанные справочники и сессии (если есть). Сюда попадают операции регистрации, логина, профиля, истории действий пользователя.
- Shard B (Books): таблица 'books' + каталоги, жанры, рейтинги, отзывы (если будут). Сюда идут тяжёлые выборки по каталогу, поиск, фильтрация, агрегации.
- Shard C (Stores): таблица 'stores/ + адреса, графики, остатки по магазинам (если будут). Сюда попадают операции по точкам продаж, маршрутизация заказов, распределение по складам.

#### Горизонтальный шардинг (разделение строк внутри одной таблицы)
Суть: одна и та же таблица разбивается на части (шарды) по ключу (user_id, book_id, store_id). Каждый шард — отдельная база/сервер.

##### Выбор ключа шардирования:

- Для users: лучший ключ — user_id. Можно использовать диапазон (user_id 1–1M → Shard 1, 1M–2M → Shard 2) или хеш от ID.
- Для books: ключ — book_id (или isbn). Часто используют хеш-шардинг, чтобы равномерно распределить нагрузку.
- Для stores: ключ — store_id. Поскольку магазинов обычно меньше, можно использовать диапазонный шардинг или даже не шардировать горизонтально, а держать несколько копий (replication) для отказоустойчивости.
##### Режимы работы серверов:

- 1. Read/Write Master: каждый шард имеет свой мастер-сервер, куда идут все записи и тяжёлые чтения.
- 2. Read Replicas: для каждого шарда создаются 1–3 реплики только для чтения. Балансировщик направляет туда SELECT-запросы.
- 3. Failover: при падении мастера одна из реплик автоматически становится мастером (через MHA, Orchestrator, Patroni или встроенные HA-решения).
- 4. Proxy/Router: на уровне приложения или через промежуточный слой (ProxySQL, Vitess, MaxScale) запросы направляются в нужный шард по ключу шардирования.

#### Как совместить вертикальный и горизонтальный шардинг
На практике делают так:

- Сначала делают вертикальный шардинг: 'users', 'books', 'stores' разнесены по разным базам.
- Затем каждую из этих баз горизонтально шардируют по своему ключу:
* ('users') → шарды по диапазонам user_id или хешу.
* ('books') → шарды по book_id.
* ('stores') → либо шардинг по store_id (если магазинов очень много), либо репликация без горизонтального шардинга (если их немного).
Пример распределения:

| Логическая сущность | Вертикальный шард | Горизонтальные шарды (пример) |
|----------------------|-------------------|-------------------------------|
| Пользователи         | DB_Users          | Shard_Users_01 (user_id 0–999k), Shard_Users_02 (1M–1.9M), … |
| Книги                | DB_Books          | Shard_Books_01, Shard_Books_02, … (по book_id) |
| Магазины             | DB_Stores         | Shard_Stores_01 (store_id 0–499), Shard_Stores_02 (500–999), … или просто реплики |

##### Блоксхема (текстовое описание)
\[Приложение / API]
\       |
\       +---> [Router / Proxy (ProxySQL / Vitess / MaxScale)]
\               |
\               +--- [DB_Users] --- [Shard_Users_01 (RW Master)] --- [Replica_Users_01]
\               |                  + [Shard_Users_02 (RW Master)] --- [Replica_Users_02]
\               |
\               +--- [DB_Books] --- [Shard_Books_01 (RW Master)] --- [Replica_Books_01]
\               |                  + [Shard_Books_02 (RW Master)] --- [Replica_Books_02]
\               |
\               +--- [DB_Stores] --- [Shard_Stores_01 (RW Master)] --- [Replica_Stores_01]
\                                  + [Shard_Stores_02 (RW Master)] --- [Replica_Stores_02]


##### Что где располагается:

- Приложение/API: знает, как формировать запросы и какой ключ шардирования использовать.
- Router/Proxy: по ключу (user_id, book_id, store_id) определяет целевой шард и направляет запрос.
- DB_Users / DB_Books / DB_Stores: логические группы баз (вертикальный шардинг).
- Shard_*: физические инстансы (серверы/контейнеры), каждый содержит часть строк таблицы.
- RW Master: принимает INSERT/UPDATE/DELETE и синхронные чтения.
- Replica: принимает SELECT, снижает нагрузку на мастера, служит для отказоустойчивости.

##### Режимы работы серверов
###### 1.Нормальный режим:

* Все мастера активны, реплики синхронизируются.
* Запросы на запись идут на мастера.
* Запросы на чтение балансируются между репликами и частично на мастера.

###### 2.Режим деградации (падение одной реплики):

* Реплика исключается из балансировки.
* Нагрузка перераспределяется на оставшиеся реплики.

###### 3.Режим failover (падение мастера):

* Система мониторинга/HA-инструмент обнаруживает недоступность мастера.
* Одна из реплик переводится в режим RW (становится новым мастером).
* Маршрутизация обновляется (Proxy/Router начинает слать запросы на новый мастер).
* После восстановления старого мастера он становится репликой или отдельным шардом.

###### 4.Режим ребалансировки (добавление нового шарда):

* Данные перераспределяются между шардами (например, часть пользователей переносится на новый шард).
* Временно может быть повышенная нагрузка на чтение/запись и сетевые операции.
